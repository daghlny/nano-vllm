# nano-vllm 学习笔记

一份持续更新的文档，记录读这个仓库过程中的学习思路和思考。

## 整体学习思路

核心原则：**理解设计为主，细节按需深入**。

nano-vllm 是一个专门写来"被读懂"的项目（1200 行，刻意简化），但这不意味着要逐行啃。判断一个细节"值不值得记"的关键，是看它属于下面哪一类。

### 两种细节，价值不同

**类型 A：能从设计决策重新推导出来的实现细节**

典型例子：`prepare_decode` 里 `slot_mapping` 的计算

```python
slot_mapping.append(seq.block_table[-1] * self.block_size + seq.last_block_num_tokens - 1)
```

只要记住抽象概念 —— **paged KV cache 把 K/V 按固定大小 block 切块、block 之间不连续、slot 是全局线性 index** —— 这个公式随时能现推。记硬公式没意义。

**类型 B：反映真正设计取舍的细节**

这类值得记，因为背后是 tradeoff：

- **为什么 CUDA Graph 只录 decode、不录 prefill？**
  Prefill 的 shape 变化太大、kernel 本身够大不缺 launch overhead；decode 反之。
- **为什么 Scheduler 严格 prefill 优先于 decode、不混批？**
  延迟 vs 吞吐的取舍；混批实现复杂度爆炸。
- **为什么 `RowParallelLinear` 的 bias 只在 rank 0 加？**
  All-reduce 会让 bias 被加 N 次，否则结果错误。
- **为什么 prefix cache 用 xxhash64 + 上一块 hash 折叠？**
  Hash chain 才能保证前缀的唯一性。

理解了这些，你在任何 LLM 推理引擎里都能快速定位。忘了也能通过"为什么"把自己问回来。

### 对 nano-vllm 的分层深度建议

| 模块 | 推荐深度 | 理由 |
|---|---|---|
| `llm_engine.py` | 通读 | 就 90 行，理解整个生命周期 |
| `scheduler.py` | **深读** | 调度策略是 LLM 服务的灵魂，每个分支都有 tradeoff |
| `block_manager.py` | **深读** | paged KV + prefix caching 是 vLLM 最核心的创新 |
| `model_runner.py` | 通读核心路径，跳过 shm/worker 细节 | CUDA graph capture 值得看，跨 rank 派发机制看个大概就行 |
| `attention.py` | **深读 forward**，FA 参数含义查一下 | 理解 prefill / prefill-with-prefix-cache / decode 三条路径 |
| `linear.py` (TP) | 理解 column/row 分片原则即可 | 除非要改 TP，否则不用记每个 `weight_loader` |
| `qwen3.py` | 扫一遍 | 标准 decoder transformer，没有 surprise |
| `sampler.py`, `loader.py` | 需要时再看 | 实现细节没有 surprise |

### 判断"要不要继续深入"的启发式

下面任何一条成立就值得深入某个细节：

1. **你要改这块** —— 改代码不看懂细节一定踩坑
2. **这个设计让你困惑 / 直觉觉得"不对吧"** —— 这种困惑是信号，背后往往藏着一个非显然的约束
3. **你打算给别人讲这段** —— 教学压力逼你必须 go deep
4. **性能 / 正确性出现反常** —— 比如 bench 数字和预期差 2×，这时候深挖 kernel 细节才有价值

都不满足就是"知道 A 决定了 B，具体算式需要时回来看代码" —— 这种认知比"背下所有行号"有用得多。

### 推荐的阅读顺序

如果目标是**理解 LLM 推理系统如何工作**（而非在这上面做生产改造）：

1. **Scheduler + BlockManager** —— vLLM 全家桶的灵魂，懂了这个看 vLLM V1 / SGLang / TensorRT-LLM 都会快
2. **Attention 的三条路径分叉**（prefill / prefill-with-cache / decode） —— 把 FA 的 `flash_attn_varlen_func` 和 `flash_attn_with_kvcache` 参数含义搞清楚
3. **CUDA Graph 和 TP** —— 工程优化层面的内容，前两步打好基础后再看会"哦原来如此"

### 可以不记的东西

- 具体的 index / offset 计算（`slot_mapping`、`last_block_num_tokens - 1` 里的 -1 为什么在那）
- `warmup_model` 用 `[0] * seq_len` 作 dummy 输入
- `SharedMemory(2**20)` 为什么是 1 MiB
- 各种 config 的默认值

这些是实现选择，不是设计知识。

### 一句话总结

**记"为什么这样设计"，现推"具体怎么算"。**

## 技术架构设计上的选择问题

记录这个项目里几个关键的"为什么这样设计"的思考。这两个问题有个共同的根，最后一节会串起来。

### 1. 为什么 CUDA Graph 只录 decode、不录 prefill？

本质是 **CUDA Graph 对"输入形状"的刚性要求，碰上 prefill 内在的形状多变性**。四层原因叠加：

**1. Shape 组合爆炸**

`prepare_prefill`（`model_runner.py:129`）传给 FA 的 metadata 每个维度都会变：

```python
input_ids       # shape = [total_tokens]，1 到 max_num_batched_tokens(16384)
cu_seqlens_q    # shape = [num_seqs + 1]，1 到 max_num_seqs(512)
max_seqlen_q    # Python int！不是 tensor
max_seqlen_k    # Python int！不是 tensor
```

组合下来是 `O(16384 × 512)` 级别的 shape 空间。CUDA Graph 要求固定 shape，即使离散化（如 `total_tokens ∈ {128, 256, 512, ...}` × `num_seqs ∈ {1, 2, 4, ...}`）也要录 70+ 张 graph，每张都要 warmup + capture，开销大、显存占用大，还要做 padding。

对比 decode：只有 **batch size** 一个维度变（`seqlen_q` 恒为 1，`seqlen_k` 通过 `cache_seqlens` tensor 传入不影响 graph shape），只需一维离散化（`[1, 2, 4, 8, 16, 32, …, 512]`），14 张 graph 覆盖全。

**2. Launch overhead 占比完全不同**

粗略量级：

| 阶段 | 单 step GPU 计算 | CPU launch overhead | 占比 |
|---|---|---|---|
| Prefill (1K tokens) | 数十 ms | 几百 μs | 1–3% |
| Decode (bs=16) | 几 ms | 几百 μs | 10–30% |
| Decode (bs=1) | ~1 ms | 几百 μs | **30–50%** |

Qwen3-0.6B 每层 ~10 个 kernel launch × 28 层 = ~280 次 launch，每次 ~5 μs → ~1.4 ms 纯 CPU 开销。Prefill 的 kernel 本身耗时几毫秒级别，launch 微不足道；decode 小 batch 下 kernel 可能只跑几十微秒，1.4 ms launch 就是瓶颈 —— 这时 CUDA Graph 用一次 `cudaGraphLaunch` 把 280 次 launch 摊到几微秒。

**一句话：prefill 是 compute-bound，launch overhead 微不足道；decode 小 batch 是 launch-bound，非 Graph 不可。**

**3. Prefill 有 runtime 分支**

Attention 契约：

- Prefill **无** prefix cache：`flash_attn_varlen_func(q, k, v, ...)` 直接在投影 K/V 上算
- Prefill **有** prefix cache：同样 `flash_attn_varlen_func`，但 K/V 从 paged cache 读、传 `block_table`

这是一个依赖 runtime state（`context.block_tables is None`？）的 if-else。CUDA Graph 捕获的是固定的 kernel launch 序列，没法录分支。要录就得每条分支各录一份，进一步放大组合数。Decode 永远走 `flash_attn_with_kvcache` 单一路径，没有分支。

**4. 附带：capture 本身很慢**

每个 bs 要 warmup 一次 + capture 一次（`model_runner.py:242-244`）。录 14 张 decode graph 在 A100 上 10–30 秒；录几十张 prefill graph 要数分钟。

### 2. 为什么 Scheduler 严格 prefill-or-decode、不混批？

这个决策直接被上面的约束推动，再叠加几个工程/工作负载上的考虑。

**1. 根本原因：prefill 和 decode 用的是不同的 FA kernel**

```python
if context.is_prefill:
    out = flash_attn_varlen_func(q, k, v, cu_seqlens_q, cu_seqlens_k, ...)
else:
    out = flash_attn_with_kvcache(q, k_cache, v_cache, cache_seqlens, block_table)
```

`flash_attn_varlen_func` 处理"一堆不等长 seq 拼起来 + cu_seqlens 切分"；`flash_attn_with_kvcache` 处理"每个 seq 1 个 Q token + 去 paged cache 读长 K/V"。完全不同的内存布局和 kernel 参数。

混批要么一 step 调两次 attention（复杂度翻倍、延迟增加），要么统一成一个 kernel（FA 后来的 varlen+paged KV 路径，vLLM V1 / SGLang 在用，但 nano-vllm 故意不拉进来）。

**2. CUDA Graph 互斥**

Prefill 不能录 graph（见上节）。一旦 step 混了 prefill，整个 step 只能走 eager，decode 部分也享受不到 graph 加速。严格分开的话，每个纯 decode step 都能吃到 graph 收益。

**3. 工作负载瓶颈资源不同**

- Prefill：矩阵乘法体积大，**计算密集**（SM 饱和）
- Decode：每 token 过一遍模型，K/V cache 大量 HBM 读取，**带宽密集**（SM 利用率低）

混批时 SM 和 HBM 都在被争用，不如分开跑效率高：
- 纯 prefill → SM 吃满、HBM 中等
- 纯 decode → HBM 吃满、SM 中等
- 混批 → 两边都"不上不下"

**4. 实现简单性收益巨大**

`Scheduler.schedule()` 返回 `(scheduled_seqs, is_prefill)` —— **一个 bool 就描述了整个 step 的模式**：

- `model_runner.run` 走 `prepare_prefill` 还是 `prepare_decode`
- `set_context` 的 `is_prefill`
- Attention 的代码分叉
- 是否走 CUDA Graph

混批要把这个 bool 变成 `List[bool]`、合并 `prepare_*`、按 seq 分类构造 `slot_mapping`、按 seq 分类调 attention。nano-vllm 会从 1200 行膨胀到 2000+。作为教学项目，保持 pure step 纯粹性比那点 throughput 重要。

**5. 代价：TTFT / ITL 取舍**

不是免费的：

- Prefill 优先 → 正在生成的 seq 感受到 **ITL 突刺**
- 不混批 → 新请求 TTFT 取决于能否抢到这个 step 的 prefill 机会

**缓解：chunked prefill**（`scheduler.py:35`）。单条 seq prefill 塞不进 token budget 就拆成多块，避免超长 seq 独占多 step 饿死 decode。vLLM v0.5 左右引入的关键优化，nano-vllm 完整保留。

真·vLLM V1 更进一步支持 mixed batching（同 step 内 prefill + decode 共享 attention kernel），靠 FA 统一的 varlen + paged KV 接口。代表"简单性"被"吞吐/延迟收益"压倒的拐点。

**和上一个问题的联系：这两个决策共享一个根**

> **CUDA Graph 只支持固定 shape + 无分支 + 单一 kernel 路径**
>    → 只有 decode 满足
>    → 为了不丢失 CUDA Graph 加速，step 必须 "纯 decode" 或 "纯 prefill"
>    → Scheduler 不混批

再叠加 attention kernel 两套 API、工作负载瓶颈资源不同、代码复杂度取舍，形成了"prefill-first + 不混批 + 仅 decode 用 graph"这一套彼此自洽的设计。真·vLLM V1 要突破这套，整体架构复杂度跳了一档。
